## 2024-06-15 - [CRITICAL] Remote Code Execution via Sympy
**Vulnerability:** Arbitrary code execution in `/api/nonlinear/reldeg` where user-provided expressions were passed directly to `sympy.sympify()`. Since `sympy.sympify` internally calls `eval()`, an attacker could pass malicious Python code like `eval('__import__("os").system(...)')` to compromise the server.
**Learning:** `sympy.sympify()` is fundamentally unsafe for untrusted input. The standard `parse_expr` even with restrictions can still be bypassed. A dedicated AST-based validator must be used to thoroughly inspect the expression tree before any evaluation.
**Prevention:** I implemented a `safe_sympify` wrapper that leverages Python's `ast.parse` to traverse the abstract syntax tree and explicitly block attributes (to prevent access to `os.system` or dunders) and strictly allowlist builtins like `abs` and `max`. Always wrap `sympy.sympify()` calls with an AST-validator when dealing with user input.

## 2024-05-24 - [HIGH] Overly Permissive CORS
**Vulnerability:** The FastAPI backend had CORS configured with `allow_origins=["*"]` while `allow_credentials=True`.
**Learning:** This combination allows any attacker website to make authenticated cross-origin requests by having the browser reflect the Origin header. Fastapi/Starlette explicitly forbids this in newer versions.
**Prevention:** Hardcode sensible defaults for development (e.g., localhost) and use environment variables (e.g., `CORS_ORIGINS`) to allow operators to securely define allowed origins in production. Never combine `*` with credentials.

## 2024-05-24 - [HIGH] Denial of Service via Large Matrix Computations
**Vulnerability:** The computational endpoints for linear systems (`/api/vstar`, `/api/ddp`, `/api/simulate`) did not validate the dimensions of the input matrices `A`, `B`, `C`, and `E`. An attacker could submit extremely large matrices (e.g., 2000x2000), causing the server to consume excessive CPU resources and tie up worker threads, leading to a Denial of Service (DoS).
**Learning:** Computational algorithms, especially those involving matrix operations (like SVD, pseudoinverses, or iterative intersections), have time complexities that scale poorly (often O(n^3)). Unbounded inputs directly translate to unbounded CPU usage.
**Prevention:** Always enforce strict boundary limits on the dimensions of input data before processing them with heavy algorithms. I implemented a Pydantic `model_validator` to limit matrix dimensions to 100x100.
## 2024-03-26 - Matrix Input Dimension DoS Bypass
**Vulnerability:** The application-layer input validator logic inside `MatrixInput` and `LinearSystemInput` restricted 2D matrix sizes by checking only the length of the first row (`len(self.matrix[0]) <= max_dim`). A payload shaped like `[[], [1]*1000000]` bypassed this check entirely, passing massive inputs to computationally expensive NumPy operations, leading to Denial of Service (DoS) and excessive memory exhaustion.
**Learning:** Checking dimensions of iterable-of-iterables relies on the structural uniformity of the payload. Pydantic's `List[List[float]]` ensures type correct elements, but not structural uniformity (i.e. all lists do not have to be the same size). This lack of structural constraint can be leveraged to bypass naive length assertions targeting only index 0.
**Prevention:** Always validate all nested elements of iterables. Use `any(len(row) > max_dim for row in matrix)` or `all(len(row) <= max_dim for row in matrix)` to ensure constraints are enforced for every row, irrespective of matrix uniformity.

## 2024-06-12 - [HIGH] Denial of Service via Massive Symbolic Computation
**Vulnerability:** The `safe_sympify` function prevented arbitrary code execution but allowed structurally valid math operations that cause computationally explosive parsing in `sympy`, such as huge exponents like `x**1000000`. Evaluating these causes a CPU and memory Denial of Service. In addition, the AST parser allowed all non-builtin function names.
**Learning:** AST whitelisting must cover computational bounds in addition to code-execution checks. Large constants in exponents (`ast.BinOp` with `ast.Pow`) must be restricted. Furthermore, non-builtin functions shouldn't be blindly permitted; only a specific whitelist of mathematical functions (e.g., `sin`, `cos`) should be allowed when parsing mathematical expressions.
**Prevention:** Always walk the AST to explicitly reject specific operations like `ast.Pow` where `right.value` is an extremely large constant, and maintain a strict explicit whitelist for function names (e.g. `ALLOWED_MATH_FUNCS`).

## 2024-06-15 - [HIGH] AST Whitelisting Bypass leading to Denial of Service
**Vulnerability:** The application-layer `safe_sympify` validator in `api/engine/nonlinear.py` checked if `ast.Pow` exponents (i.e., `node.right`) were `ast.Constant` values exceeding 100 to prevent computational DOS. However, if an attacker provided a non-constant expression as the exponent, like `x**(10*10*10*10*10)` (where `node.right` is an `ast.BinOp`), the `isinstance` check failed silently. The expression was deemed safe and passed to SymPy, where it evaluated to massive exponents leading to CPU and memory exhaustion.
**Learning:** Security checks that selectively inspect node types without enforcing a strict whitelist or schema on the *expected* types are prone to bypasses. If an AST visitor only restricts large constants but fails to reject non-constant structures, it allows attackers to build complex structures that circumvent the specific check.
**Prevention:** Always fail securely by enforcing a strict schema. When parsing an exponent (`ast.Pow`), explicitly require that the right side is an `ast.Constant` and reject any other node type (like `ast.BinOp`, `ast.Name`, etc.) with a `ValueError` before checking its value.

## 2024-06-15 - [HIGH] ReDoS via Exponential Complexity Bypass in SymPy
**Vulnerability:** The application-layer `safe_sympify` validator in `api/engine/nonlinear.py` blocked explicitly large constant exponents (like `x**1000`), but failed to evaluate or limit the complexity of non-constant exponent structures, such as `x**(y**z)`. Attackers could bypass the constant-value checks and submit deep or highly complex AST expressions to `ast.Pow`, causing SymPy to hang indefinitely, resulting in a Denial of Service.
**Learning:** Checking for purely bounded constant values is insufficient when dealing with arbitrary algebraic parser engines. Any component of a mathematical expression tree that dictates operational complexity (such as exponents) must be recursively validated not just for explicit value limits, but also for AST depth and allowed operand combinations to prevent explosive evaluation combinations.
**Prevention:** When validating exponents (`ast.Pow`), use a recursive AST check on the right node to ensure it only contains bounded constants (<= 100), simple variables (`ast.Name`), or basic arithmetic combinations, strictly capping AST depth. This prevents computationally explosive nested variable powers that bypass simple constant checks.

## 2024-10-27 - [HIGH] ReDoS via Exponential Complexity Bypass in SymPy using ast.BinOp
**Vulnerability:** The application-layer `safe_sympify` validator in `api/engine/nonlinear.py` blocked explicitly large constant exponents (like `x**1000`) and deeply nested `ast.Pow` inside other `ast.Pow`, but allowed other `ast.BinOp` inside `ast.Pow`. This allows users to pass structurally complex exponential forms like `x**(y**z)` or any `ast.BinOp` evaluating to a massive number inside an exponent.
**Learning:** Checking for purely bounded constant values is insufficient when dealing with arbitrary algebraic parser engines. When an AST visitor restricts the depth or specific structures but fails to explicitly whitelist the allowed structures or strictly restrict internal node types, attackers can use alternative node types to trigger the same exponential complexity bypass in `sympy`.
**Prevention:** Explicitly restrict allowed nested exponent operations by throwing `ValueError` when `isinstance(n.op, ast.Pow)` inside the AST BinOp recursive check for exponent constraints. Enforce an explicit node whitelist.

## 2025-02-19 - [HIGH] Denial of Service via Unbounded Variable Name Length
**Vulnerability:** The `NonlinearSystemInput` Pydantic model correctly restricted the number of variables, vector field components, and mathematical expression lengths, but failed to restrict the length of individual variable names within the `vars` array. This allowed an attacker to send requests containing single variable names that are millions of characters long, leading to memory exhaustion and Application-Layer Denial of Service (DoS) during validation or parsing.
**Learning:** Limiting the number of elements in an array or the length of explicit "long string" fields (like mathematical expressions) is insufficient if other string fields (like variable names) are left unbounded. Attackers can exploit any unbounded variable to exhaust system memory. Structural uniformity checks or total payload limits are necessary, but explicit bounds on all string components provide defense-in-depth.
**Prevention:** Always enforce a maximum length on all string fields, especially when those strings represent identifiers or variable names. Added a check `any(len(v) > max_var_len for v in self.vars)` to explicitly restrict variable name sizes.

## 2024-11-20 - [HIGH] Denial of Service via SymPy Polynomial Expansion
**Vulnerability:** The application-layer `safe_sympify` validator correctly blocked large constants and nested exponentiation but allowed complex polynomial structures with modest exponents, such as `(x1+x2+x3+x4+x5+x6+x7+x8+x9+x10)**50`. When passed to `sympy.expand()` in `compute_relative_degree()`, this triggered an exponential expansion of terms, leading to excessive memory allocation, CPU exhaustion, and Application-Layer Denial of Service (DoS).
**Learning:** Limiting explicitly massive numbers (e.g., exponents > 100) is insufficient when the underlying computational engine expands algebraic terms. Expanding a polynomial like `(x + y)**n` produces an exponentially growing number of terms proportional to the base complexity and exponent size. Even seemingly small exponents (e.g., 50) on multivariable bases hang the parser indefinitely during expansion.
**Prevention:** Strictly bound the permitted exponent values for algebraic computations. The maximum exponent permitted in `ast.Pow` inside `safe_sympify` was reduced from 100 to 10, preventing computationally explosive expansions while remaining sufficient for typical educational control theory problems.
## 2025-02-19 - [HIGH] ReDoS via Explicit Node Whitelisting Bypass in SymPy using Unrestricted BinOp
**Vulnerability:** The application-layer `safe_sympify` validator in `api/engine/nonlinear.py` blocked nested `ast.Pow` inside exponents, but allowed any other binary operations inside an exponent. This allowed an attacker to use any `ast.BinOp` (like bitwise shifts `ast.LShift`) that may be unexpectedly computationally expensive or evaluated unsafely, triggering CPU and memory exhaustion in the parsing/evaluating phase.
**Learning:** It is not enough to explicitly blacklist known dangerous nodes (like `ast.Pow`) inside a recursive check if the parser allows multiple structural forms. You must strictly whitelist only the operations that are demonstrably safe for the use case (e.g. `ast.Add`, `ast.Sub`, `ast.Mult`, `ast.Div`).
**Prevention:** Always fail securely by enforcing an explicit whitelist. When validating exponents, ensure that `isinstance(n.op, (ast.Add, ast.Sub, ast.Mult, ast.Div))` is explicitly checked, raising a `ValueError` for any unsupported binary operations inside the exponent.

## 2026-04-09 - [HIGH] ReDoS via Explicit Node Whitelisting Bypass in SymPy using Unrestricted BinOp (Global)
**Vulnerability:** The application-layer `safe_sympify` validator in `api/engine/nonlinear.py` explicitly restricted unsupported binary operations (`ast.LShift`, `ast.RShift`, etc.) *inside* an exponent (`ast.Pow`), but failed to enforce this same explicit node whitelist across the entire Abstract Syntax Tree (AST). This allowed an attacker to bypass the complexity checks by placing unsupported and potentially expensive operations anywhere outside of an exponent (e.g. `x << y`), triggering exceptions or resource exhaustion.
**Learning:** Securing a parser engine requires a globally applied zero-trust model. Restricting a vulnerable subset of a data structure (like just the exponent of a power operation) is insufficient if the rest of the structure is evaluated with a permissive implicit-allow model.
**Prevention:** Apply a global whitelist constraint. Within the AST walker in `safe_sympify`, any `ast.BinOp` node encountered is now strictly verified against an explicit whitelist (`(ast.Add, ast.Sub, ast.Mult, ast.Div, ast.Pow)`), failing securely if an unsupported operation is found anywhere in the expression.
## 2026-04-10 - [Medium] Security Enhancement: Missing Security Headers
**Vulnerability:** The Next.js frontend was missing the `Content-Security-Policy` header and exposed the `X-Powered-By: Next.js` header, which could leak information about the technology stack and leave the application vulnerable to XSS attacks (Defense in Depth).
**Learning:** Next.js allows injecting security headers via the `headers()` async function in `next.config.js`. However, it does not set `Content-Security-Policy` by default, and `X-Powered-By` must be explicitly disabled via `poweredByHeader: false`.
**Prevention:** Always verify that `next.config.js` sets a robust `Content-Security-Policy` in its `securityHeaders` array and explicitly includes `poweredByHeader: false` in the `nextConfig` object.

## 2026-04-14 - [Medium] Security Enhancement: Missing Backend Security Headers
**Vulnerability:** The FastAPI backend did not set any HTTP security headers in its API responses. If an API endpoint is accessed directly in a browser or framed, it could be susceptible to MIME-sniffing or clickjacking attacks (Defense in Depth).
**Learning:** Security headers should be applied at all layers. While the Next.js frontend has its own security headers in `next.config.js`, the Python FastAPI backend was missing them.
**Prevention:** Always verify that backend APIs use middleware (like `BaseHTTPMiddleware` in FastAPI) to explicitly set security headers such as `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Strict-Transport-Security`, and `Content-Security-Policy`.
